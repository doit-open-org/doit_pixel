# Device-to-Controller Communication Protocol - BLE

Communication between a device and its controller primarily supports the following:

- Discovery and control before cloud communication is established.
- Faster control response when the network environment is poor.
- Control while a router has lost Internet access but remains powered on.
- Local communication for devices that do not support cloud connections.

## Communication Model

`Note: LAN communication for Wi-Fi products is supplementary only and does not require heartbeats. A successful connection and send operation is considered online; no connection or a send failure is considered disconnected.`

## Communication Flow

Device:

    Provision Wi-Fi, connect to the mobile phone, register with the server, and wait for data communication.

Controller:

    Guide the user into pairing mode, discover the device, bind it, and operate it.

## Communication Conventions

### GAP Advertising Format

**Advertising packet (advertising data)**

| Advertising data field | Type | Description |
| :---: | :---: | :--- |
| Device LE physical connection flag | `0x01` | Length: `0x02`; type: `0x01`; data: `0x06` |
| Service UUID | `0x02` | Length: `0x03`; type: `0x02`; data: `0x01A2` |
| Service Data | `0x16` | Length: `0x11`; type: `0x16`<br>Data:<br>Service name: `01 A2`<br>`protocol_ver`: protocol version<br>`dev_type`: device connection type<br>`pid`: unique identifier<br>`mac`: device MAC address |

Advertising data example (10-byte device ID): `02 01 06 03 02 01 A2 11 16 01 A2 08 01 01 00 00 00 00 00 00 00 00 00 00 00 00`

```c
// Little-endian byte order
typedef struct {
    unsigned int8 link_flag_l;
    unsigned int8 link_flag_t;
    unsigned int8 link_flag_d;

    unsigned int8 service_uuid_l;
    unsigned int8 service_uuid_t;
    unsigned int8 service_uuid_dh;
    unsigned int8 service_uuid_dl;

    unsigned int8 service_data_l;
    unsigned int8 service_data_t;
    unsigned int8 service_name_h;
    unsigned int8 service_name_l;

    unsigned int8 protocol_ver;
    unsigned int8 dev_type;
    unsigned int8 pid[6];
    unsigned int8 dev_mac[6];
} doit_protocol01_adv_t;
```

**Scan response packet (scan response data)**

| Advertising data field | Type | Description |
| --- | --- | --- |
| Complete Local Name | `0x09` | Length: `0x03`; type: `0x09`; data: `0x54`, `0x54` |
| Manufacturer-specific data | `0xff` | Length: `0x19`<br>Type: `0xff`<br>Data:<br>Binding flag: `0x00` (bit 1 is the binding flag; other bits are reserved)<br>Device key: a 10-byte key generated by the app and stored as the server-side `dev_id`; used as a shared secret for app-to-BLE communication<br>`pid`: generated by the platform<br>`mac`: device MAC address<br>`end`: reserved bytes for OTA-related use |

Advertising data example (unbound): `03 09 54 54 19 FF 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00`

```c
// Little-endian byte order
typedef struct {
    unsigned int8 dev_name_len; // 03
    unsigned int8 dev_name_t;   // 09
    unsigned int8 dev_name_h;   // ASCII 'T' (54)
    unsigned int8 dev_name_l;   // ASCII 'T' (54)

    unsigned int8 data_len;     // 19
    unsigned int8 data_t;       // FF
    unsigned int8 binded_flag;  // 0: unbound; 1: bound
    unsigned int8 dev_id[10];   // Device ID
    unsigned int8 pid[6];       // 17 18 19 20 21 22
    unsigned int8 dev_mac[6];   // 23 24 25 26 27 28

    unsigned int8 end[2];       // Reserved, OTA-related
} doit_protocol01_rsp_data_t;
```

### GATT Communication Format

Bluetooth 4.0 can send only 20 bytes per frame. To support Bluetooth 4.0 chips, this protocol uses 20-byte frames. Since command packets can easily exceed 20 bytes, larger packets are fragmented. Note the following:

- Send all fragments of one large packet before sending the next packet.
- The total packet length must not exceed 256 bytes.
- If data exceeds 256 bytes, implement application-level fragmentation markers such as sequence numbers within the payload.

## Request Packet (All Writes)

| Offset<br>(byte) | Length<br>(byte) | Content<br>(hex) | Description |
| :---: | :---: | :---: | :---: |
| 0 | 1 | `01` | Protocol version |
| 1 | 1 | `pkg_sn` | Packet sequence number |
| 2 | 17 | `data` | Data |
| 19 | 1 | `CRC8` | CRC-8 over every byte except the checksum |

## Response Packet (All Reads)

| Offset<br>(byte) | Length<br>(byte) | Content<br>(hex) | Description |
| :---: | :---: | :---: | :---: |
| 0 | 1 | `01` | Protocol version |
| 1 | 1 | `pkg_sn` | Packet sequence number |
| 2 | 17 | `res`, `data` | Response and data |
| 19 | 1 | `CRC8` | CRC-8 over every byte except the checksum |

```c
// Little-endian byte order: request frame
typedef struct {
    unsigned int8 protocol_ver;
    unsigned int8 pkg_sn;
    void          data[17];
    unsigned int8 crc8;
} doit_protocol01_req_data_t;
```

```c
// Little-endian byte order: response frame
typedef struct {
    unsigned int8 protocol_ver;
    unsigned int8 pkg_sn;
    void          data[17];
    unsigned int8 crc8;
} doit_protocol01_rsp_data_t;
```

`The most significant bit of pkg_sn indicates whether more bytes follow. A value of 1 means no more bytes follow. Verify that preceding data is complete, then apply timeout handling and reset the sequence counter.`

## Device Communication (Cloud & LAN)

### CMD: 0, Query Device Information

```c
// Little-endian byte order: device information request
typedef struct {
    unsigned int8 cmd;
    unsigned int8 len = 4;
    unsigned int8 timestamp[4];
} doit_protocol01_devinfo_req_t;

// Little-endian byte order: device information response
typedef struct {
    unsigned int8 cmd;
    unsigned int8 len = 31;
    unsigned int8 dev_id[10]; // 4-byte timestamp + 6-byte MAC
    unsigned int8 dev_type;   // Wi-Fi = 0x00; BLE = 0x01; Wi-Fi + BLE = 0x02
    unsigned int8 software_ver[4];
    unsigned int8 hardware_ver[4];
    unsigned int8 dev_mac[6];
    unsigned int8 pid[6];
```

### CMD: 1, Send Provisioning Information

**Sent by the controller to the device:**

```c
// Little-endian byte order: provisioning information request
typedef struct {
    unsigned int8 cmd;
    unsigned int8 len = 163;
    // A named SSID indicates a binding packet for a Wi-Fi-capable device.
    // 32 bytes of 02 indicate a BLE binding packet.
    // 32 bytes of 01 indicate BLE connection validation (key).
    unsigned int8 ssid[32];
    unsigned int8 passwd[64];
    unsigned int8 bssid[6];
    unsigned int8 dev_key[10]; // Assigned by the app
    unsigned int8 open_id[33];
    unsigned int8 lat[4];
    unsigned int8 lng[4];
    unsigned int8 domain[10];
} doit_protocol01_wifiinfo_req_t;

// Little-endian byte order: provisioning information response
typedef struct {
    unsigned int8 cmd;
    unsigned int8 len = 1;
    unsigned int8 res;
} doit_protocol01_wifiinfo_rsp_t;
```

**Response (`res`) values**

| `res` | Description |
| :---: | :--- |
| 0 | Success |
| 1 | Invalid data or key |
| 2 | The device is already bound and provisioning; validation is rejected |
| 3 | The device is not bound; validation is rejected |

**Sent by the controller to the device:**

After receiving this command, the device must send an unbind request to the server, delete stored Wi-Fi or BLE information and user information, then restart in provisioning mode.

### CMD: 5, OTA Upgrade

**Sent by the controller to the device:**

```c
// Sends an OTA command and enters OTA mode immediately.
typedef struct {
    unsigned int8 cmd;
    unsigned int8 len;
    unsigned int8 pkg_index_h; // High byte of OTA packet number: (pkg_index_h << 8) | pkg_index_l
    unsigned int8 pkg_index_l; // Low byte of OTA packet number
    unsigned int8 data[len - 2];
} doit_protocol01_ota_t;

// Response after 17 small packets are received
typedef struct {
    unsigned int8 cmd;
    unsigned int8 len = 1;
    unsigned int8 res;
} doit_protocol01_res_t;
```

`res = 0` means the device can continue the OTA upgrade.

`res = 1` means the OTA upgrade must restart.

## Device Provisioning Flow

### Bluetooth Provisioning Flow

- The app enables Bluetooth and receives advertising frames.
- It obtains the device ID, binding flag, and device key from the advertising frames, then reports them to the server.
- The server response determines whether the device is valid. A valid device with no binding and no device key is shown in the UI. If the device is bound and its device key matches the one stored by the app, the app makes a second connection and sends CMD 1 again for matching. A response of 0 means the second connection succeeded; 1 means it failed.
- After the user selects a device, the app connects to it, queries device information to create the corresponding control UI, and then issues the device key.
- Once connected to the network, the device submits a login request to the server with the provisioning device key and proactively reports status to the server.
- The app polls the device online status until timeout. If the device becomes online, provisioning is considered successful.
- On success, leave the provisioning flow and open the device control screen. On failure, guide the user to retry and analyze or report the cause.

## Device OTA Flow

Based on the device and server versions, the app must display an upgrade prompt. Silent upgrades are not allowed.
